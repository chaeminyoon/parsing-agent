# 골든셋 (Human-Labeled Ground Truth)

파이프라인의 자체 점수가 사람 판단과 얼마나 맞는지 검증하기 위한 데이터셋이다. 라벨이 쌓이기 전까지 이 디렉토리는 스캐폴드(가이드 + 스키마 + 평가 스크립트)만 담는다.

## 왜 필요한가

지금 파이프라인 점수는 전부 자체 메트릭 + LLM judge다. "0.93"이 사람 눈에도 좋은 문서인지 검증된 적이 없다. 골든셋이 생기면 세 가지가 가능해진다.

1. **메트릭 검증** — 자체 점수와 사람 점수의 상관(Spearman)이 낮은 메트릭은 고치거나 버린다
2. **judge 캘리브레이션** — judge 점수를 사람 점수 분포에 맞게 보정한다 (isotonic/Platt)
3. **회귀 게이트** — 프롬프트나 수리 로직을 바꿨을 때 사람 기준 품질이 떨어졌는지 CI에서 잡는다

## 라벨링 절차

1. 문서마다 **평가자 2명이 독립적으로** 라벨링한다 (서로 라벨을 보지 않는다)
2. 파이프라인이 만든 `문서.md`를 원본 PDF와 나란히 놓고 아래 항목을 매긴다
3. 두 평가자의 항목별 일치도(가중 Cohen's kappa)를 계산한다 — **0.7 미만이면 라벨링 기준을 다시 합의하고 재라벨링**
4. 불일치 항목은 3번째 평가자가 중재한다

### 라벨 항목

| 필드 | 값 | 기준 |
|---|---|---|
| `overall_quality` | 1~5 | 이 마크다운을 RAG 인덱스에 그대로 넣어도 되는가. 5=수정 불필요, 3=부분 수정 필요, 1=사용 불가 |
| `text_coverage` | 1~5 | 원본 본문이 얼마나 남았나. 5=누락 없음, 3=문단 단위 누락, 1=절반 이상 누락 |
| `table_correctness` | 1~5 | 표의 구조(행/열/병합)와 값이 맞나. 표가 없는 문서는 5 |
| `structure_preservation` | 1~5 | 장/절 헤딩, 목록, 순서가 원본과 일치하나 |
| `hallucination` | true/false | 원본에 없는 내용이 생성됐나 (LLM 수리·비전 복구 검증용) |
| `notes` | 자유 텍스트 | 판정 근거. 특히 3점 이하 항목은 필수 |

점수 앵커를 문장으로 고정해 두는 이유: "3점"의 의미가 평가자마다 다르면 kappa가 낮게 나오고 데이터셋 전체가 무효가 된다.

## 파일 구조

```
golden/
  README.md            # 이 문서
  labels/
    schema.json        # 라벨 JSON 스키마
    example.json       # 작성 예시
    <문서명>.json       # 평가자별 라벨 (rater 필드로 구분, 같은 파일에 append)
  run_golden_eval.py   # 파이프라인 점수 vs 사람 라벨 상관 분석
```

문서 원본 PDF는 저장소에 커밋하지 않는다 (`data/`는 gitignore). 라벨 파일에는 문서 파일명만 기록한다.

## 평가 실행

```bash
# 1. 골든 문서들을 파이프라인에 돌린다
uv run python -m parsing_agent.cli "data/문서.pdf" --output-dir outputs/golden-run

# 2. labels/ 에 라벨을 작성한다 (schema.json 참고)

# 3. 상관 분석
uv run python golden/run_golden_eval.py --run-dir outputs/golden-run --labels golden/labels
```

출력: 평가자 간 일치도(kappa), 메트릭별 사람 점수와의 Spearman 상관, judge 점수 캘리브레이션 산점 데이터. 라벨이 5건 미만이면 상관계수 대신 개별 대조표만 출력한다 (표본이 작을 때 상관계수는 노이즈다).

## 목표 규모

- 1차: 20건 (메트릭별 상관의 방향이라도 확인)
- 2차: 100건 (judge 캘리브레이션 가능, 표 복잡도·스캔 품질별 층화 표집)
